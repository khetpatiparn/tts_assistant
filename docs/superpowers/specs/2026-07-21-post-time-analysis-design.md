# Design: วิเคราะห์เวลาโพสต์

วันที่: 2026-07-21
สถานะ: อนุมัติ scope แล้ว รอ writing-plans

## ปัญหา

ผู้ใช้อยากรู้ว่า **เวลาที่โพสต์คลิปสัมพันธ์กับยอดวิวไหม** — เพราะถ้าใช่ นี่คือการปรับที่ไม่มีต้นทุนเลย
(ไม่ต้องทำคลิปเพิ่ม ไม่ต้องจ่ายเงิน แค่เปลี่ยนเวลากดโพสต์) ซึ่งสำคัญมากเมื่อข้อมูลจริงชี้ว่า
**วิวคือ lever ไม่ใช่ conversion** (ดู `2026-07-21-recommender-design.md` — conv rate ทั้งช่องเกาะกลุ่ม
0.06–0.29% แต่วิวต่างกัน 60 เท่า)

**แต่ตอนนี้ไม่มีที่ไหนบันทึกเวลาโพสต์เลย** — `PromptEntry.postedAt` เก็บเป็น UTC midnight (วันที่ล้วน)
และ `Content.csv` ของ TikTok ก็ให้แค่วันที่ ("July 1") ไม่มีชั่วโมง

## ข้อค้นพบจากการสำรวจ (load-bearing — ฐานของทุกการตัดสินใจ)

### 1. Video ID ของ TikTok ฝังเวลาไว้อยู่แล้ว

TikTok video ID เป็น snowflake — `Number(BigInt(id) >> 32n) * 1000` = Unix ms ของ **เวลาอัปโหลด**
ทดสอบกับ 15 คลิปจริงแล้ว วันที่ถอดได้ตรงกับที่บันทึกไว้ 13/15 และเวลาที่ได้สมเหตุสมผลทั้งหมด
(10:00–22:00 ไม่มีเวลามั่วแบบตี 3 ซึ่งถ้าเทคนิคผิดจะเห็นกระจายสุ่ม)

**แปลว่าไม่ต้องให้ผู้ใช้กรอกอะไรเลย และวิเคราะห์ย้อนหลังได้ทันทีทุกคลิปที่เคยโพสต์**

### 2. ความคลาดเคลื่อนของคลิปโพสต์สด: ~21 วินาที (ไม่มีผล)

จากคลิปที่ TikTok แจ้งเวลาโพสต์เป๊ะมาในหนังสือแจ้งละเมิด:
`ID = 23:16:30` vs `TikTok แจ้งว่าโพสต์ 23:16:51` → ต่าง **21 วินาที**
ไม่มีผลกับการวิเคราะห์ระดับชั่วโมง (วัดได้จากคลิปเดียวเพราะมีเคสนี้เคสเดียวที่รู้เวลาเป๊ะ
แต่สอดคล้องกับกลไก: โพสต์สด = อัปแล้วเผยแพร่ทันที)

### 3. คลิปตั้งเวลาทำให้ ID บอกเวลาผิด — แต่ตรวจจับได้

ID ถูกสร้างตอน **อัปโหลด** ไม่ใช่ตอน **เผยแพร่** คลิปที่ตั้งเวลาไว้จึงได้เวลาผิด

**วิธีตรวจจับ:** เทียบ *วันจาก ID* กับ *วันเผยแพร่ใน `Content.csv`* — ถ้าไม่ตรง = ตั้งเวลาแน่นอน

ผลจริง เจอ 2 จาก 15:

| คลิป | อัปโหลด (ID) | เผยแพร่ (ไฟล์) | ห่าง |
|---|---|---|---|
| **ที่ลับเล็บแมว** (คลิปทำเงินหลัก) | 28 มิ.ย. 19:11 | 1 ก.ค. | 3 วัน |
| ชั้นวางเครื่องปรุง | 20 มิ.ย. 18:47 | 26 มิ.ย. | 6 วัน |

**นี่คือหลักฐานว่าถ้าใช้ ID อย่างเดียวจะพังจริง** — คลิปที่ทำเงินมากที่สุดคือหนึ่งในตัวที่ข้อมูลจะผิด

**และกู้คืนไม่ได้** — ไม่มีแหล่งใดให้ชั่วโมงเผยแพร่ของคลิปตั้งเวลา (ยืนยันแล้วว่า TikTok oEmbed
ไม่มีฟิลด์เวลาใดๆ เลย — คืนแค่ title/author/thumbnail)

### 4. FollowerActivity.csv มีข้อมูลผู้ชมออนไลน์รายชั่วโมง

`creative_data/Followers_rainny0192/FollowerActivity.csv` — คอลัมน์ `Date, Hour, Active followers`
167 แถว = 7 วัน (14–20 ก.ค.) × 24 ชม. **ตอนนี้ระบบไม่ได้อ่านไฟล์นี้เลย**

ค่าเฉลี่ยรายชั่วโมง: พีค **20:00 (210)**, 21:00/19:00 (201), พีคย่อยเที่ยง (193) · ต่ำสุด 02:00–03:00 (52)
→ ห่างกัน ~4 เท่า

**เทียบกับพฤติกรรมจริง:** คลิปโพสต์สด 13 ตัว กระจุกที่ **10:00–13:00 (9 ใน 13)** มีแค่ 3 ตัวที่โพสต์เย็น
→ **ผู้ใช้อาจกำลังโพสต์สวนช่วงพีคของผู้ชมตัวเอง** (ยังฟันธงไม่ได้ ดูข้อ 5)

⚠️ ข้อควรระวัง: "ผู้ติดตามออนไลน์" บน TikTok สำคัญน้อยกว่าแพลตฟอร์มอื่น เพราะวิวส่วนใหญ่มาจาก FYP
(คนไม่ได้ติดตาม) ใช้เป็น **สมมติฐานประกอบ ไม่ใช่คำตอบ**

### 5. ตัวอย่างน้อยเกินกว่าจะฟันธง

ใช้ได้จริงแค่ **13 คลิป** (โพสต์สด + มีข้อมูลวิว) และวิวถูกครอบงำโดย outlier 2 ตัว (63k, 17.8k)
**ระบบต้องไม่สรุปว่า "โพสต์เวลานี้ดีที่สุด"** จากข้อมูลเท่านี้ — จะเป็นระบบที่ดูน่าเชื่อแต่เป็น noise
(โรคเดียวกับ `CLIP_AWAITING_THRESHOLD` ที่เพิ่งถอดออกไป)

## ขอบเขต

**ทำ:** ถอดเวลาจาก video ID · toggle โพสต์สด/ตั้งเวลาตอนวางลิงก์ (default ปรับตามพฤติกรรม) ·
ติดป้ายที่มาของเวลา · ตรวจสอบข้ามกับ `Content.csv` · import `FollowerActivity.csv` ·
section แสดงผลใน Dashboard

**ไม่ทำ:** ฟันธงว่าควรโพสต์เวลาไหน · ถามเวลาย้อนหลังกับคลิปเก่า · แตะ `postedAt` เดิม ·
ทำนายวิวจากเวลา

## สถาปัตยกรรม

### 1. รูปแบบเวลาทั่วทั้งฟีเจอร์: **24 ชั่วโมงเท่านั้น**

ทุกจุดที่แสดงเวลาต้องเป็น 24 ชม. — ช่องกรอก, แกนกราฟ (`00:00`–`23:00` ไม่ใช่ `12 AM`/`1 PM`),
ข้อความสรุป (เช่น "โพสต์บ่อยสุดช่วง 10:00–13:00")

หมายเหตุ: `<input type="time">` เก็บค่าเป็น `"HH:MM"` 24 ชม. เสมอในระดับข้อมูลอยู่แล้ว ส่วนการแสดงผล
ขึ้นกับ locale ของเบราว์เซอร์ (เครื่องผู้ใช้เป็นไทย จึงแสดง 24 ชม.) — แต่ทุกที่ที่ **เรา** เรนเดอร์เอง
ต้องบังคับ 24 ชม.

เวลาทั้งหมดคำนวณเป็น **เวลาไทย (UTC+7)** เพราะผู้ชมเป็นคนไทย (TH 58.6% จาก `FollowerTopTerritories.csv`)

**⚠️ จุดที่พลาดแล้วเพี้ยน 7 ชั่วโมง — ต้องทำให้ตรงกันทั้งสองแหล่ง:**

| แหล่ง | ฐานเวลาที่ได้มา | ต้องแปลงยังไง |
|---|---|---|
| **ชั่วโมงที่โพสต์** (จาก video ID) | Unix timestamp = **UTC** | **ต้อง +7 ชม.** ก่อนหาเลขชั่วโมง |
| **ชั่วโมงผู้ติดตามออนไลน์** (`FollowerActivity.csv`) | TikTok Studio export ตามโซนเวลาบัญชี = **เวลาไทยอยู่แล้ว** | **ใช้เลข `Hour` ดิบได้เลย ห้ามแปลงซ้ำ** |

ถ้าเผลอแปลง `Hour` ของ FollowerActivity อีกรอบ กราฟสองเส้นจะเหลื่อมกัน 7 ชั่วโมงแบบเงียบๆ —
ดูเผินๆ เหมือนใช้ได้ แต่ข้อสรุปจะผิดทั้งหมด ตอน implement ให้ตรวจด้วยข้อมูลจริง: พีคของ
FollowerActivity ต้องอยู่ที่ **20:00** ถ้าออกมาเป็น 03:00 แปลว่าแปลงเกิน

### 2. Data model

**`PromptEntry` เพิ่ม 2 ฟิลด์:**

```prisma
  postedTimeSource String?   // "derived" | "manual" | null
  isScheduledPost  Boolean   @default(false)
```

- `postedTimeOfDay` (มีอยู่แล้วจาก plan ก่อน, ยังว่างทั้ง 49 แถว) → **นำมาใช้จริงในฟีเจอร์นี้**
  เก็บ `"HH:MM"` เฉพาะกรณี `manual`
- `postedTimeSource` = ป้ายบอกที่มา (หัวใจของความน่าเชื่อถือ)
- `isScheduledPost` = ผู้ใช้ติ๊กว่าตั้งเวลา (หรือระบบตรวจเจอทีหลัง)
- **ห้ามแตะ `postedAt`** — เก็บเป็น UTC midnight, ผูกกับ `lib/entry-sort.ts` + date helper สองตัวใน
  `production-panel.tsx` ที่ห้ามสลับกัน

**ตารางใหม่ `FollowerActivity`:**

```prisma
model FollowerActivity {
  id         String   @id @default(cuid())
  activityOn DateTime          // วันที่ (เวลาไทย, เก็บเป็น UTC midnight)
  hour       Int               // 0-23
  active     Int
  importedAt DateTime @default(now())

  @@unique([activityOn, hour])
}
```

`@@unique([activityOn, hour])` = idempotent ต่อการ import ไฟล์เดิมซ้ำ (แพทเทิร์นเดียวกับ `ClipMetric`)

### 3. ที่มาของเวลา — 3 ระดับ

| ป้าย | ได้มายังไง | ความแม่น | ใช้ในการวิเคราะห์ |
|---|---|---|---|
| `derived` | ถอดจาก video ID (โพสต์สด) | ~21 วินาที | ✅ ใช้เป็นค่าเริ่มต้น |
| `manual` | ผู้ใช้กรอกตอนวางลิงก์ | แม่นถ้ากรอกตอนสด | ⚙️ toggle เปิด/ปิดได้ในหน้าผล |
| `null` | ตั้งเวลา + ไม่ได้กรอก | ไม่รู้ | ❌ ข้าม |

**เหตุผลที่ต้องติดป้าย:** ถ้าเอาเวลาแม่นกับเวลาประมาณมาปนกันโดยแยกไม่ออก ข้อสรุปอาจถูกขับเคลื่อน
โดยข้อมูลที่แต่งขึ้นทั้งหมดโดยไม่มีใครรู้ — ที่ตัวอย่างเพียง 13 ตัว การเติมข้อมูลเดา 3 ตัวคือ
**19% ปนเปื้อน** พอจะพลิกสัญญาณอ่อนๆ ได้

การมีป้ายทำให้ตอบได้ว่า **"ถ้ารวมข้อมูลที่กรอกเองด้วย ผลเปลี่ยนไหม"** → เปลี่ยนเยอะ = อย่าเชื่อ,
ไม่เปลี่ยน = มั่นใจได้ นั่นคือประโยชน์จริงของมัน ไม่ใช่แค่เก็บ metadata

### 4. ตอนวางลิงก์ — toggle ที่ปรับ default เอง

ใน `production-panel.tsx` ข้างช่อง video URL เพิ่ม toggle: **`โพสต์สด` | `ตั้งเวลา`**

- **default = แบบที่ผู้ใช้ทำบ่อยที่สุด** คำนวณจากประวัติ (`isScheduledPost` ของ entry ที่มีลิงก์แล้ว)
  ปัจจุบัน 13/15 = โพสต์สด → default = โพสต์สด → **ผู้ใช้ไม่ต้องกดอะไรเลยในเคสปกติ**
  ถ้าวันหนึ่งพฤติกรรมพลิก default สลับเอง (หลักการเดียวกับเกณฑ์ self-scaling อื่นในโปรเจกต์นี้)
- เลือก `โพสต์สด` → ไม่ต้องกรอกอะไร ระบบใช้เวลาจาก ID (`postedTimeSource = "derived"`)
- เลือก `ตั้งเวลา` → โผล่ช่อง `<input type="time">` · กรอกก็ได้ ไม่กรอกก็ได้ (**ไม่บังคับ ไม่ทวง**)
  - กรอก → `postedTimeOfDay` + `postedTimeSource = "manual"`
  - ไม่กรอก → `postedTimeSource = null` → ถูกข้ามในการวิเคราะห์

**กฎเหล็ก: ห้ามชวนให้เดาย้อนหลัง** — ระบบจะไม่มีวันขึ้นช่องถามเวลาของคลิปเก่าที่วางลิงก์ไปนานแล้ว
เพราะคำตอบจะเป็นการเดา และการเดาที่ถูกบันทึกจะกลายเป็น "ข้อเท็จจริง" ในสายตาระบบตลอดไป

### 5. ตรวจสอบข้ามตอน import Content.csv

หลัง import แล้ว เทียบ **วันจาก video ID** กับ **`postedDate` ใน `ClipMetric`**:

- ตรงกัน → ยืนยันโพสต์สด เวลา `derived` เชื่อได้
- **ไม่ตรง แต่ entry ติ๊กว่าโพสต์สด** → ผู้ใช้ติ๊กผิด → ตั้ง `isScheduledPost = true` และ
  **ตัดออกจากการวิเคราะห์เอง** (ไม่เอาเวลาปลอมเข้าไปปน)

นี่คือเหตุผลที่ toggle ผิดพลาดได้ไม่เป็นไร — ข้อมูลจริงจับได้ทีหลังเสมอ **ระบบไม่พึ่งความถูกต้อง
ของสิ่งที่ผู้ใช้ติ๊ก**

### 6. Parser — `lib/follower-activity.ts`

ไฟล์เดียวที่รู้จักโครง `FollowerActivity.csv` (แพทเทิร์นเดียวกับ `lib/clip-metrics.ts`)
- CSV มี BOM · ทุก field ครอบ quote · คอลัมน์: `0=Date` `1=Hour` `2=Active followers`
- `Date` เป็น `"July 14"` ไม่มีปี → เดาปีจากวันที่ import (ใช้ helper แบบเดียวกับ `parseAsOfDate`
  ใน `lib/clip-metrics.ts` — **ควร reuse ไม่เขียนซ้ำ**)

### 7. การวิเคราะห์ — `lib/post-time.ts`

pure function ล้วน ไม่เรียก LLM ไม่แตะ DB (แบบเดียวกับ `lib/recommender.ts` / `lib/dashboard.ts`)

```ts
export type PostTimeSource = "derived" | "manual";

export type HourBucket = {
  hour: number;          // 0-23 เวลาไทย
  clipCount: number;
  medianViews: number;   // median กัน outlier 63k บิดเบือน
  followersActive: number | null;
};

export type PostTimeAnalysis = {
  buckets: HourBucket[];       // 24 ช่องเสมอ (ชั่วโมงที่ไม่มีคลิป = clipCount 0)
  usableClips: number;
  skippedScheduled: number;
  scheduledRatio: number;
  enoughData: boolean;
  clipsNeededForConfidence: number;
};
```

**ใช้ median ไม่ใช่ mean** สำหรับวิว — เพราะ 63k/17.8k จะลาก mean จนอ่านไม่ได้ความ

`enoughData` = `usableClips >= 30` (ตอนนี้ 13 → ยังไม่ถึง) เกณฑ์นี้เป็นค่าคงที่ที่ผูกกับสเกลข้อมูล
→ ต้องบันทึกไว้ในหัวข้อ "ค่าคงที่ที่ผูกกับสเกลข้อมูล" ของ CLAUDE.md ตามแบบแผนที่ตั้งไว้

### 8. UI — section ใหม่ใน Dashboard

`components/post-time-panel.tsx` แสดงกราฟแกนชั่วโมง **00:00–23:00** ซ้อน 2 ชั้น:
- **แท่ง**: median วิวของคลิปที่โพสต์ชั่วโมงนั้น (+ จำนวนคลิป)
- **เส้น**: ผู้ติดตามออนไลน์เฉลี่ยชั่วโมงนั้น

ใช้ **Recharts** (v3.9.2 — API ต่างจากตัวอย่าง v2 ทั่วไป ต้องเช็ก `node_modules/recharts/types/` จริง
ก่อนเขียน) สีจาก design token เดิมผ่าน `var(--color-*)`

**บรรทัดสถานะที่ต้องซื่อสัตย์เสมอ:**
> วิเคราะห์จาก 13 คลิป · ข้าม 2 (ตั้งเวลา) · **ยังน้อยเกินจะสรุป** — เก็บอีก ~17 คลิปจะเริ่มเชื่อได้

- toggle "รวมเวลาที่กรอกเอง" (default = ปิด) → เปิดแล้วเห็นว่าผลเปลี่ยนไหม
- เตือนเมื่อ `scheduledRatio > 0.4`: *"คลิปตั้งเวลาเยอะขึ้นจนข้อมูลเริ่มไม่พอ — ถ้าอยากให้วิเคราะห์ต่อได้
  ต้องเริ่มกรอกเวลาเผยแพร่ตอนวางลิงก์"*
- **ห้ามเขียนข้อความชี้นำแบบ "ควรโพสต์ 20:00"** จนกว่า `enoughData` เป็น true
