# Entry Dates & Sidebar Navigation — Design

วันที่: 2026-07-12
Branch: `feature/entry-dates-and-sidebar-nav`

## ปัญหา

ผู้ใช้กำลังจะไล่กรอกข้อมูลคลิปเก่า ~30 อันที่ลงไปแล้วตลอดเดือนที่ผ่านมา (ลงวันละ 1 คลิป) เข้าระบบ ซึ่งเปิดปัญหา 2 อย่างที่เกี่ยวพันกัน

### 1. ไม่มีวันที่ลงคลิปจริง

`PromptEntry.createdAt` สแตมป์ตอน**กดสร้าง entry ในระบบ** ไม่ใช่ตอนลงคลิป เพราะฉะนั้นถ้าไล่กรอกคลิปเก่าทั้ง 30 อันวันนี้ ทุกอันจะได้ `createdAt` = วันนี้เหมือนกันหมด

ผลที่ตามมาทันที: `history-rail.tsx` เรียงรายการตาม `createdAt` (ผ่าน `orderBy` ใน `app/page.tsx`) และคำนวณเลข Take จากลำดับนั้น → **ลำดับใน sidebar จะกลายเป็นลำดับที่พิมพ์เข้ามา ไม่ใช่ลำดับที่ลงคลิปจริง** เลข Take (T01, T02, ...) ก็มั่วตามไปด้วย

วันที่ลงคลิปยัง**หาย้อนหลังได้** (ดูจากคลิปบน TikTok) แต่ต้นทุนต่างกันมาก: กรอกตอนนี้ที่กำลังเปิดคลิปดูอยู่แล้ว = ไม่กี่วินาทีต่ออัน / กลับมาหาทีหลัง = ต้องเปิดลิงก์ทีละ 30 อันใหม่

### 2. Sidebar ใช้งานไม่ไหวเมื่อรายการเยอะ

ปัจจุบัน sidebar เป็นลิสต์ยาวเรียงเดียว ไม่มีค้นหา ไม่มีกรอง พอมี 30+ รายการ (และจะเพิ่มวันละอัน = ปีละ ~365) การหา entry ที่ต้องการต้องเลื่อนไล่สายตาอย่างเดียว

แยกเป็น 2 ปัญหาย่อยที่ต่างกัน:
- **ตอนไล่กรอกข้อมูลเก่า:** ต้องรู้ว่าทำไปถึงไหน เหลืออันไหนที่ยังไม่ได้กรอกผลลัพธ์ — เป็นการ**ติดตามความคืบหน้า** ไม่ใช่การค้นหา
- **ระยะยาว:** "คลิปถ้วยกาแฟเดือนก่อนใช้ prompt อะไร" — เป็นการ**ค้นหา** ในกองที่โตขึ้นเรื่อยๆ

## ขอบเขต

### ทำ

**1. `postedAt` — วันที่ลงคลิป TikTok จริง**
- ฟิลด์ใหม่ใน `PromptEntry`, optional (entry ที่ยังไม่ได้ลงคลิปเว้นว่างได้)
- ช่องกรอกวันที่ในแท็บ ② (ผลลัพธ์ & คลิป) ถัดจากลิงก์ TikTok
- Sidebar เรียงตาม `postedAt` ใหม่→เก่า โดย entry ที่ยังไม่มีวันที่อยู่บนสุด (เป็นงานที่ยังทำไม่เสร็จ ควรเห็นก่อน)

**2. Sidebar: ค้นหา / จัดกลุ่ม / กรอง**
- **ช่องค้นหา** บนสุด — กรองรายการตามชื่อสินค้า (client-side, พิมพ์แล้วกรองทันที)
- **จัดกลุ่มตามเดือน** — หัวข้อคั่นตามเดือนของ `postedAt` (เช่น "ก.ค. 2026") ให้มีจุดยึดสายตาเวลาเลื่อน; entry ที่ไม่มี `postedAt` รวมอยู่กลุ่ม "ยังไม่ได้ลงคลิป" บนสุด
- **ตัวกรอง "ยังไม่ได้กรอกผลลัพธ์"** — ปุ่มสลับโหมด แสดงเฉพาะ entry ที่ยังไม่มี `chatgptOutput` หรือ `videoUrl` ครบ; ใช้ติดตามความคืบหน้าตอนไล่กรอกข้อมูลเก่า พอกรอกครบลิสต์จะว่าง = จบงาน

### ไม่ทำ (ตัดสินใจแล้ว)

- **ไม่แตะช่องยอดวิว** — มีอยู่แล้ว ปล่อยไว้เหมือนเดิม ผู้ใช้จะกรอกหรือไม่กรอกก็ได้ (ยอดวิวย้อนดูจากลิงก์ได้ตลอด และเทียบข้าม entry ไม่ได้อยู่ดีเพราะคลิปอายุไม่เท่ากัน)
- **ไม่ทำ dashboard / กราฟ / วิเคราะห์อัตโนมัติ** — ยังไม่มี label ว่าอะไรเวิร์ค และสินค้ากลบผลของ prompt อยู่ (1 สินค้า = 1 คลิป = 1 prompt ทำให้แยกผลของ prompt ออกจากผลของสินค้าไม่ได้) การวิเคราะห์ที่มีความหมายต้องเปลี่ยนวิธี**ลงคลิป** (A/B สินค้าเดียวกันหลายคลิป) ไม่ใช่แค่เปลี่ยนวิธีเก็บข้อมูล
- **ไม่ทำ pagination / virtual scroll** — ที่ระดับหลักร้อยรายการ การ render ทั้งลิสต์แล้วกรองด้วย JS ยังเร็วพอ ไม่ต้อง optimize ล่วงหน้า

## โครงสร้างข้อมูล

```prisma
model PromptEntry {
  // ...ของเดิมทั้งหมด...
  postedAt DateTime?   // วันที่ลงคลิป TikTok จริง (ว่างได้ = ยังไม่ได้ลง)
}
```

Optional เพื่อให้ entry เดิมที่มีอยู่ไม่พัง และให้กรอกย้อนหลังแบบทยอยทำได้

**การเรียงลำดับ:** `app/page.tsx` เปลี่ยน `orderBy` เป็นเรียงตาม `postedAt` (desc, nulls first) แล้วตกมาที่ `createdAt` (desc) เป็นตัวตัดสินรอง — SQLite/Prisma รองรับ `{ postedAt: { sort: "desc", nulls: "first" } }` ถ้าใช้ไม่ได้ให้ดึงมาแล้วเรียงฝั่ง JS แทน (รายการหลักร้อย ไม่กระทบ performance)

**เลข Take:** ปัจจุบันคำนวณจาก index ในลิสต์ที่เรียงตาม `createdAt` เมื่อเปลี่ยนไปเรียงตาม `postedAt` เลข Take จะสะท้อนลำดับการลงคลิปจริง ซึ่งเป็นความหมายที่ถูกต้องกว่าเดิม

## หน้าจอ

### แท็บ ② ผลลัพธ์ & คลิป (`production-panel.tsx`)

เพิ่มช่องวันที่ (`<Input type="date">`) วางคู่กับลิงก์ TikTok ในแถวเดียวกัน — ทั้งสองอย่างเป็นข้อมูลของ "คลิปที่ลงไปแล้ว" เหมือนกัน

### Sidebar (`history-rail.tsx`)

```
┌──────────────────────────┐
│ [+ สินค้าใหม่]            │
│ [ค้นหาสินค้า...        ] │  ← ช่องค้นหา
│ [ ] ยังไม่ได้กรอกผลลัพธ์  │  ← ตัวกรอง (toggle)
│                          │
│ ยังไม่ได้ลงคลิป           │  ← หัวข้อกลุ่ม
│   T31 หมอนรองคอ          │
│ ─────────────────        │
│ ก.ค. 2026                │  ← หัวข้อกลุ่ม (เดือน)
│   T30 ถ้วยกาแฟ  12 ก.ค.  │
│   T29 ไม้ถูพื้น  11 ก.ค.  │
│ ─────────────────        │
│ มิ.ย. 2026               │
│   T28 ...                │
└──────────────────────────┘
```

- ค้นหา/กรอง/จัดกลุ่ม ทำเป็น logic ฝั่ง client ทั้งหมดใน `history-rail.tsx` ไม่ต้อง query DB เพิ่ม
- แสดงวันที่แบบสั้น (เช่น "12 ก.ค.") ใต้/ข้างชื่อสินค้า
- state ของช่องค้นหาและตัวกรองอยู่ใน `history-rail.tsx` เอง (ไม่ต้องยกขึ้นไป `prompt-workspace.tsx` เพราะไม่มีใครอื่นใช้)
- เมื่อค้นหา/กรองแล้วไม่เจออะไร แสดงข้อความว่าง (เช่น "ไม่พบรายการ") แทนลิสต์เปล่า

## การทดสอบ

ไม่มี test runner ในโปรเจกต์ ใช้วิธีเดิม: `npm run build` (type-check ในตัว) + `npm run lint` แล้วขับผ่าน Playwright

1. `npx prisma migrate dev` + `npx prisma generate`
2. entry เดิมที่ไม่มี `postedAt` ยังเปิดได้ปกติ ไม่ crash และไปอยู่กลุ่ม "ยังไม่ได้ลงคลิป"
3. กรอก `postedAt` ในแท็บ ② → บันทึก → รีโหลด → ค่ายังอยู่ และ entry ย้ายไปอยู่กลุ่มเดือนที่ถูกต้อง
4. Sidebar เรียงตาม `postedAt` ใหม่→เก่า โดยอันที่ไม่มีวันที่อยู่บนสุด
5. ช่องค้นหา: พิมพ์ชื่อสินค้าบางส่วน → เหลือเฉพาะรายการที่ตรง; ล้างค่า → กลับมาครบ
6. ตัวกรอง "ยังไม่ได้กรอกผลลัพธ์": เปิด → เห็นเฉพาะ entry ที่ยังไม่มี `chatgptOutput` หรือ `videoUrl`; กรอกครบแล้ว entry นั้นหายจากลิสต์
7. ไม่มี console error
